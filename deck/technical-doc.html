Go Language Support in OpenRewrite:
Technical Architecture

This document describes how we built automated code remediation for Go — a language that has never had it before. It covers the architecture of the Go language module in OpenRewrite, how it extends the existing Java-native platform to support a language with fundamentally different semantics, how the template matching system works, and where the technology is headed.

The audience is anyone who wants to understand what's actually happening under the hood when a recipe transforms Go source code. Not the marketing version. The engineering version.

Why Go Is Different

Every mainstream language except Go has a code transformation ecosystem. Java has OpenRewrite and Error Prone. C# has Roslyn analyzers with code fixes. Python has Ruff with auto-fix. TypeScript has codemods and ESLint's --fix flag. These tools don't just find problems — they fix them. They produce deterministic, reviewable diffs that can be applied across thousands of repositories in a single operation.

Go has staticcheck and golangci-lint. Both are excellent at finding problems. Neither fixes them. Every diagnostic is a line item for a human: triage, understand the context, make the edit, run the tests, open a PR. Multiply that by the number of findings, multiply again by the number of repositories, and you have a remediation backlog that grows faster than any team can burn it down.

The reason Go doesn't have automated remediation isn't that nobody wanted it. It's that building it requires solving a specific set of hard problems. You need a lossless syntax tree that preserves every whitespace character and comment. You need a visitor infrastructure that lets you walk and modify that tree without corrupting formatting. You need a type system mapping that bridges Go's structural typing to the refactoring primitives the platform already has. And you need all of this to round-trip through an RPC bridge so Go code can be transformed by recipes running inside a Java-native platform.

That's what we built.

Architecture: How Go Extends OpenRewrite

The Lossless Semantic Tree

OpenRewrite's core abstraction is the Lossless Semantic Tree (LST). Unlike a conventional AST, an LST preserves every character of the original source file — whitespace, comments, blank lines, trailing commas. When you print an unmodified LST, you get back the exact original source, byte for byte. This property is what makes OpenRewrite recipes trustworthy: the diff shows exactly what the recipe changed and nothing else. No spurious formatting noise. No reformatting wars.

For Go, we built an LST that maps Go's syntax onto the same J (Java-derived) node hierarchy that OpenRewrite uses for every language. A Go function declaration maps to MethodDeclaration. A Go for loop maps to ForLoop or ForEachLoop. A method call maps to MethodInvocation. This isn't just naming convention — it means that cross-language recipes and visitors that operate on the J interface work for Go without modification. A recipe that renames a method invocation works identically whether the underlying language is Java, C#, Python, or Go.

Where Go has constructs that don't exist in Java — goroutines, channels, defer statements, multiple return values, short variable declarations, composite literals, select statements — we extend the tree with Go-specific node types: GoStmt, Channel, Defer, Send, MultiAssignment, Composite, CommClause, Slice, MapType, and others. Each carries its own whitespace preservation and integrates cleanly with the visitor dispatch infrastructure.

The Go LST currently defines 45+ node types. Every node carries a Prefix (the whitespace and comments before the node), Markers (metadata like search results or diagnostics), and type-specific fields. For example, a MethodInvocation has a Select (the receiver or package), Name (the method name), Arguments (the argument list), and MethodType (the resolved type signature from Go's type checker).

The Parser: From Go Source to LST

The Go parser wraps Go's standard library packages — go/parser, go/ast, go/types, and go/token — to produce the OpenRewrite LST. The standard library handles the hard parsing work; our code handles the whitespace-preserving mapping.

This is where things get subtle. Go's go/ast produces an abstract syntax tree that discards whitespace. Our parser maintains a cursor that walks through the original source text in lockstep with the AST traversal. When the AST says "here's an identifier at position 42", our parser extracts everything between the cursor and position 42 as the prefix whitespace, advances the cursor past the identifier, and continues. Comments are parsed out of the whitespace into structured Comment objects with their own suffix whitespace. The result is a tree where every byte of the original source is accounted for — either as node content or as whitespace in a Space object.

The parser also performs type attribution using go/types. Every identifier, expression, and method invocation gets its resolved type attached as a JavaType object. This uses the same type hierarchy as Java's type system — JavaTypeClass, JavaTypeMethod, JavaTypePrimitive, JavaTypeParameterized — with Go-specific mappings. Go's int maps to a primitive. Go's named types map to classes. Go's structural interfaces map to class types with interface kind. Go's multiple return values (which have no Java equivalent) map to JavaTypeUnknown — an area for future improvement.

The RPC Bridge: Go Inside a Java Platform

OpenRewrite is a Java platform. The Moderne CLI and Moderne Platform are Java applications. But Go code needs to be parsed by a Go parser — you can't parse Go accurately in Java. The solution is an RPC bridge.

When the Moderne CLI encounters a Go repository, it launches a Go subprocess (the rewrite-go RPC server) and communicates with it over JSON-RPC. The Java side sends "parse this file" requests; the Go side parses the source, builds the LST, and sends it back as a serialized tree. The Java side then runs recipes against the tree using the same visitor infrastructure it uses for every other language.

The RPC layer handles the serialization of every LST node type, every whitespace object, every type attribution, and every marker. It also handles bidirectional communication — when a recipe needs to visit a tree node that was received from Go, and the node references other nodes by ID, the Java side can request those nodes from the Go process.

This architecture means Go recipes can be written in Go (running in the Go subprocess) or in Java (running in the platform). The recipe infrastructure we've built — GoTemplate, GoPattern, MethodMatcher — runs natively in Go. The platform orchestration — recipe discovery, marketplace, multi-repository execution — runs in Java. The RPC bridge makes them a single system.

The Visitor Infrastructure

OpenRewrite recipes work by visiting LST nodes. The visitor walks the tree depth-first, calling a type-specific method for each node: VisitMethodInvocation, VisitBinary, VisitIf, and so on. The default implementation of each visit method recursively visits child nodes and returns the (possibly modified) node.

Go's visitor — GoVisitor — implements 45+ visit methods corresponding to the 45+ LST node types. A recipe author overrides the relevant method, calls the base implementation first (to recurse into children), then applies transformation logic. The critical invariant: always call the base visitor first. This ensures child transformations happen before parent transformations, which is essential for composing recipes.

Go's visitor uses a virtual dispatch mechanism that mirrors Java's visitor pattern in a language without inheritance. A Self field points to the outermost struct, enabling method override dispatch. The visitor.Init() function uses reflection to wire this up automatically, so recipe authors just embed GoVisitor and override methods.

We made a deliberate decision early on to ensure that the base visitor's visit methods recurse into all child nodes. This sounds obvious, but the initial implementation was incomplete — several methods visited prefix and markers but didn't recurse into child expressions. For example, VisitMethodInvocation visited the method name but not the arguments or the receiver expression. VisitVariableDeclarator visited the name but not the initializer. We fixed seven visit methods to ensure complete recursion. Without this, recipes that transform deeply nested expressions simply wouldn't fire.

The Template Matching System

The Problem Templates Solve

Writing a recipe as a manual visitor is powerful but verbose. A recipe that replaces fmt.Sprintf("%s", s) with s requires a visitor struct, a VisitMethodInvocation override, method matcher checks, argument count validation, literal value comparison, AST node construction for the replacement, and careful whitespace preservation. That's 50+ lines of Go code for what is conceptually a one-line pattern transformation.

This is the same problem Google's Refaster solved for Java. Refaster lets you write a Java class with @BeforeTemplate and @AfterTemplate methods, and an annotation processor compiles it into an OpenRewrite recipe. The recipe author writes the pattern in the target language, and the framework handles the matching and replacement.

Go doesn't have annotation processors. But it turns out we don't need one. The matching and replacement machinery can run entirely at runtime, which in some ways is more flexible than compile-time generation.

How GoTemplate and GoPattern Work

The core idea: parse a Go code string into an LST fragment, then match or substitute against it. A pattern like fmt.Sprintf("%s", __plh_s__) is a real Go expression that the parser can parse — if we declare __plh_s__ as a variable in a preamble. The placeholder identifier __plh_s__ is the capture point: when matching, it binds to whatever subtree occupies that position in the candidate code.

The scaffold system wraps template code in a compilable Go source so go/parser can parse it. For an expression template like fmt.Sprintf("%s", __plh_s__), the scaffold is:

package __tmpl__
import "fmt"
var __plh_s__ any
var __v__ = fmt.Sprintf("%s", __plh_s__)

The parser produces an LST for the full file. The scaffold extractor navigates past the package declaration, imports, and preamble variables to find the target node — the initializer expression of __v__. That extracted subtree is the pattern tree.

Matching works by structural comparison. The PatternMatchingComparator walks the pattern tree and the candidate tree in lockstep, comparing node types and structural properties. When it encounters a placeholder identifier, instead of requiring an exact match, it binds the candidate subtree to that capture name in a MatchResult. If the same capture name appears multiple times in the pattern, the comparator enforces structural equality between the two bindings — this is how you express patterns like "the same expression appears on both sides."

The comparator covers all 45+ node types. For each, it compares the semantically meaningful properties (operator kind, identifier name, literal value) while ignoring whitespace and markers. A Binary node matches if its left operand, operator, and right operand all match. A MethodInvocation matches if its receiver, method name, and arguments all match. An Identifier matches if its name matches. And a placeholder identifier matches anything, binding the candidate to the capture.

Replacement works by substitution. The GoTemplate parses the after-template the same way as the pattern, producing a template tree. The SubstitutionVisitor walks the template tree, replacing placeholder identifiers with the captured subtrees from the MatchResult. The result is a new subtree with the template's structure but the captured values from the original code.

The Leading Prefix Problem

One subtle issue deserves attention because it affects every template-based transformation. In Go's LST, compound nodes like Binary and MethodInvocation often have empty prefix whitespace. The actual leading whitespace lives on the leftmost leaf node — the first identifier or literal in the expression. This means 1 + 2 in the source var a = 1 + 2 has a Binary node with empty prefix, a Literal 1 with prefix " " (the space after =), and a Literal 2 with prefix " " (the space after +).

When a recipe replaces a Binary with a simpler expression, it needs to preserve the leading whitespace. But "the leading whitespace" isn't on the Binary node — it's on the Binary's left child. We built getLeadingPrefix and setLeadingPrefix functions that recursively walk to the leftmost leaf to get and set the effective prefix. Without these, every compound-to-simple replacement would either double the whitespace (if you naively copy the root prefix) or lose it entirely.

Multiple Before Patterns: The "anyOf" Mechanism

Google's Refaster supports Refaster.anyOf() — a way to specify multiple before-templates that all map to the same after-template. Our equivalent is calling WithBefore() multiple times in the recipe builder. The templateRecipeVisitor tries each before-pattern in order; the first match wins.

This is essential for recipes like boolean simplification. The pattern x == true should simplify to x. But so should true == x, x != false, and false != x. That's four before-patterns mapping to one after-pattern. Without "anyOf", you'd need four separate recipes or a complex manual visitor with conditional logic. With it, you get:

var x = template.Expr("x")
simplifyToKeep := template.NewRecipe(
    template.WithBefore(fmt.Sprintf(`%s == true`, x)),
    template.WithBefore(fmt.Sprintf(`true == %s`, x)),
    template.WithBefore(fmt.Sprintf(`%s != false`, x)),
    template.WithBefore(fmt.Sprintf(`false != %s`, x)),
    template.WithAfter(fmt.Sprintf(`%s`, x)),
    template.WithCaptures(x),
)

Ten lines. The framework handles the rest: scaffold parsing, pattern matching against every AST node in the file, capture binding, substitution, and whitespace preservation.

The NewRecipe Builder

The template.NewRecipe() function is the declarative recipe API. It takes functional options — WithBefore, WithAfter, WithCaptures, Imports — and produces a recipe.Recipe that can be registered in the marketplace. Under the hood, it builds GoPattern objects for each before-pattern, a GoTemplate for the after-template, and a templateRecipeVisitor that wires them together.

The builder auto-detects whether the template code is an expression, a statement, or a top-level declaration based on leading keywords. fmt.Sprintf(...) is an expression. if err != nil { return err } is a statement. func foo() {} is a top-level declaration. Each scaffold kind uses a different wrapping strategy so the Go parser can handle it. Explicit overrides (AsExpression(), AsStatement()) are available for ambiguous cases.

Template recipes compose. A recipe struct can implement RecipeList() to return multiple sub-recipes. The boolean simplification recipe, for example, is a composite of two template recipes — one for the "keep" patterns (x == true → x) and one for the "negate" patterns (x == false → !x). The framework runs sub-recipes in sequence.

Beyond Single-Node Transforms: Block-Level and File-Level Visiting

The Limitation of Node-Level Visitors

Template recipes and simple visitors operate on individual AST nodes. A visitor's VisitMethodInvocation method receives a single method call and returns a single replacement node. This works for most transformations — replacing one expression with another, simplifying a binary operation, renaming an identifier.

But some transformations require modifying the context around a node, not just the node itself. Adding a defer f.Close() after an os.Open() call requires inserting a new statement into the parent block. Hoisting a regexp.MustCompile() call out of a loop requires moving a statement from inside a loop to before it. Extracting an inline errors.New() to a package-level variable requires adding a declaration to the file's top-level statements.

These are the transformations that most linter tools can't do. They're also the transformations that deliver the most value — the ones that would take a human the most time to apply manually. We solved them by visiting at the Block level and the CompilationUnit level.

Block-Level Visiting: Statement Insertion, Hoisting, and Splicing

A Block in the LST is a brace-delimited list of statements. By visiting Block instead of individual nodes, a recipe can scan the statement list, find patterns that span multiple statements, and return a modified statement list.

Defer insertion. The "ensure file closed" recipe visits each Block, scanning for assignment statements where the right-hand side is a call to os.Open(), db.Query(), time.NewTicker(), or similar resource-creation functions. When it finds one, it extracts the variable name from the left-hand side, checks that a matching defer doesn't already exist in the block, and inserts defer f.Close() (or defer ticker.Stop(), etc.) immediately after the assignment. The recipe constructs the defer AST node with whitespace matching the surrounding code.

Regex hoisting. The "compile regex outside loop" recipe visits each Block, finds ForLoop and ForEachLoop statements, and scans their bodies for regexp.MustCompile("pattern") calls with string literal arguments. When it finds one, it creates a var compiledRegex = regexp.MustCompile("pattern") declaration, inserts it before the loop in the block's statement list, and replaces the in-loop call with a reference to the hoisted variable. For regexp.Compile (which returns two values), it promotes to MustCompile since the pattern is a compile-time literal.

strings.Builder insertion. The same Block-level pattern handles the s += expr to strings.Builder transformation. The recipe inserts var builder strings.Builder before the loop, replaces s += expr with builder.WriteString(expr) inside the loop, and inserts s = builder.String() after the loop. Three statement modifications in a single Block visit.

Redundant else removal. if cond { return } else { body } becomes if cond { return }\nbody. The recipe scans the Block's statements for If nodes where the Then block ends with a Return and an Else part is present. It removes the Else, keeps the If, and splices the else body's statements as siblings after the If in the block's statement list. A dedent sub-visitor fixes the indentation of the hoisted statements.

Guard clause refactoring. if err == nil { <deeply nested body> } becomes if err != nil { return }\n<body>. The recipe inverts the condition, inserts an early return, and promotes the body to the outer scope with corrected indentation.

CompilationUnit-Level Visiting: File-Wide Transforms

Some transforms need to modify the file as a whole. The "extract error sentinel" recipe is the best example. It visits the CompilationUnit (the entire file), runs a sub-visitor to collect all inline errors.New("message") calls inside function bodies, generates variable names from the message strings ("not found" → ErrNotFound), adds var ErrNotFound = errors.New("not found") to the top-level declarations, and replaces the inline calls with references to the new variables. Two passes over the file: one to collect, one to replace.

The "use strings.Builder in loop" recipe also visits CompilationUnit to add import "strings" when it's not already present. Import management is a CompilationUnit-level concern.

MethodMatcher and TypeUtils

MethodMatcher: AspectJ-Style Pattern Matching

Many recipes need to match specific method calls. "Does this MethodInvocation call fmt.Sprintf?" is a question that comes up constantly. You could write mi.Name.Name == "Sprintf" && mi.Select != nil && ident.Name == "fmt" every time, but that's fragile and repetitive.

MethodMatcher provides AspectJ-style pattern matching. You write "fmt Sprintf(..)" and it matches any call to fmt.Sprintf regardless of arguments. The pattern "time.Time Sub(time.Time)" matches Sub on a time.Time receiver with one time.Time argument. Wildcards work: * matches any single name, *..* matches any type in any package, .. in argument position matches any number of arguments.

The implementation compiles glob patterns to regular expressions at construction time, so matching at visit time is a regex check — fast. When type information is available (from the parser's type attribution), the matcher checks the MethodType.DeclaringType for accurate matching. When type info isn't available, it falls back to the Select expression's identifier name.

TypeUtils: Type Hierarchy Checking

Go uses structural typing for interfaces — a type satisfies an interface if it has the right methods, without an explicit implements declaration. Our TypeUtils provides type checking that works within this model: IsAssignableTo(type, "error") walks the type's interface list and supertype chain with cycle detection. IsError(type), IsString(type), IsNumeric(type) provide common checks. AsClass(type), AsMethod(type) provide safe type casts.

Type attribution coverage depends on the Go parser's ability to resolve types. For code within the parsed module, type resolution is complete. For external dependencies, it depends on whether the module cache is populated. Recipes are designed to degrade gracefully — when type info isn't available, they fall back to structural matching (identifier names, method patterns).

Markup Severity: Not Just "Found"

Recipes that can't auto-fix a pattern still need to communicate what they found and how serious it is. Our markup system provides three severity levels: MarkupWarn for definite issues (SQL injection risk, race condition, panic), MarkupInfo for suggestions (consider using X, ensure Y is closed), and MarkupError for critical findings. This replaces the generic SearchResult marker used by earlier recipe frameworks.

Severity matters because it determines how findings are triaged. A MarkupWarn on a hardcoded credential or a TLS verification bypass is a security issue that needs human review. A MarkupInfo on a context.TODO() call is a technical debt item that can wait. The markup level flows through to the Moderne Platform's UI, enabling severity-based filtering and prioritization.

Where This Is Headed

Go Version Migration

The C# recipe library has composite recipes that chain migrations: UpgradeToDotNet10 runs UpgradeToDotNet9 which runs UpgradeToDotNet8 and so on, all the way back. Each version in the chain applies the API changes, deprecation replacements, and NuGet version bumps for that specific version.

We want the same for Go. UpgradeToGo1_23 would chain through 1.22, 1.21, 1.18, applying the API adoptions at each step: generics patterns in 1.18, min/max builtins in 1.21, range-over-func in 1.22, iterator patterns in 1.23. Each composite recipe includes everything below it — you don't need to know your starting version.

Framework Migrations

The template recipe mechanism makes framework migrations straightforward to author. A migration from one HTTP router to another is largely method-to-method API mapping. gin.Context.JSON() → chi's render.JSON(). logrus.WithField() → slog.With(). The template system handles the matching and replacement; the recipe author just defines the patterns.

Module Dependency Management

Go module versioning, go.mod updates, and dependency graph resolution. When a recipe changes an API call that requires a newer version of a dependency, the module management layer updates go.mod to require the new minimum version and runs go mod tidy to resolve the graph.

Deeper Type Attribution

The biggest gap in the current implementation is type attribution completeness. Go's type checker in go/types can resolve types for code within the parsed module, but external dependencies require the module cache. As we improve parser coverage, recipes will be able to make type-dependent decisions: removing truly redundant type conversions, matching method calls by receiver type rather than just receiver name, and understanding interface satisfaction for refactoring.

Multi-File Refactoring

Renaming a method declaration also needs to rename every caller. Extracting an interface needs to update every implementation. Moving a package needs to rewrite every import. These are multi-file transformations that the Moderne Platform orchestrates — each file sees a recipe that modifies its own content, but the platform coordinates which files need which modifications.

Staticcheck/golangci-lint Harness

The C# recipe library validates every recipe against its equivalent Roslyn analyzer on real open-source repositories. We have the infrastructure for the Go equivalent — diagnostic mapping types, a working set of real repos, space validation — but the automated comparison pipeline is next. The goal: for every recipe that corresponds to a staticcheck or golangci-lint diagnostic, prove that our transformation produces the same result (or a strictly better one).

The bottom line. Go is no longer a language where linters find problems and humans fix them. The transformation infrastructure exists. The recipe library covers the most common quality, security, and modernization patterns. And it runs inside the same Moderne platform that already handles Java, C#, Python, and TypeScript — so deploying it across your Go repositories is the same workflow your teams already know.
